# CLAUDE.md

## Project Overview

This repository is a Python flight-price ingestion worker and the foundation for a future data engineering pipeline.

Current responsibilities:
- load active flight watches from PostgreSQL
- fetch flight prices from Google Flights
- persist ingestion runs and normalized offers
- evaluate alert rules
- write alert events
- run as a scheduled batch worker

This repository is **not** the main business API.
Spring Boot owns the application/business API layer.
This repository owns ingestion and monitoring workflows.

## Architecture Boundaries

### This project owns
- batch ingestion jobs
- crawling and parsing
- persistence into `ingestion.*`
- alert evaluation and event creation
- operational logging and debugging artifacts
- future DE-oriented workflows

### This project does not own
- user-facing REST APIs
- frontend concerns
- auth
- airport management UI
- app business workflows outside ingestion/monitoring

### Shared database ownership
- `ingestion.*` is owned by this Python project
- `app.*` is business-facing and primarily owned by the Spring Boot backend
- this project may read/write a limited set of `app.*` tables required for monitoring, such as:
  - `app.flight_watches`
  - `app.flight_alert_events`

## Core Principles

- Prefer simple, explicit code over clever abstractions.
- Keep ingestion code deterministic and easy to debug.
- Favor append-only historical data patterns where possible.
- Never mix raw ingestion concerns with app-facing business logic unless explicitly required.
- Keep scraping logic isolated from monitoring logic and persistence logic.
- Make code changes small and reversible.
- Always preserve observability when refactoring worker flows.

## Coding Standards

### Python
- Target Python 3.12.
- Use type hints for all new or modified functions where practical.
- Prefer dataclass or Pydantic models for structured data passed between layers.
- Keep functions short and single-purpose.
- Avoid deeply nested control flow.
- Avoid hidden side effects.

### Style
- Use Ruff formatting and linting standards.
- Line length: 100.
- Prefer double quotes.
- Prefer explicit imports over wildcard imports.
- Avoid magic strings when a named constant or enum-like value is clearer.

### Error handling
- Raise domain-specific exceptions where appropriate.
- Do not swallow exceptions silently.
- When handling external failures such as Playwright timeouts, preserve enough debug context to investigate.
- A single watch failure must not crash the whole batch if the orchestration layer can safely continue.

## Project Structure Conventions

Use these layers consistently:

- `domain/` for business and workflow models
- `database/models/` for SQLAlchemy models
- `repositories/` for persistence and read queries
- `services/` for orchestration and business logic
- `providers/` for external source integrations
- `workers/` for runnable entrypoints
- `tests/` for automated verification

### Rules
- Repositories should not contain scraping logic.
- Providers should not know about business alert rules.
- Workers should remain thin orchestration entrypoints.
- Service methods should be the main place where workflow composition happens.
- Prefer adding a new repository/service instead of making one file handle every concern.

## Database and Migration Rules

- Use SQLAlchemy models under `database/models/`.
- Use Alembic for schema changes.
- Review autogenerated migrations before applying them.
- Never trust Alembic autogenerate blindly when multiple schemas exist.
- When generating migrations, ensure that unrelated tables are not dropped.
- Keep migrations focused on one concern.

### Schemas
- `ingestion` for fetch runs and offers
- `app` for watches and alert events used by the worker

### Migration safety
Before running `upgrade head`, always verify:
- only intended tables/indexes are created or modified
- no unexpected `drop_table` or `drop_index`
- no accidental cross-schema damage

## Scraping Rules

- Google Flights selectors are unstable. Prefer resilient extraction strategies.
- Treat page structure as untrusted and likely to change.
- Navigation success and DOM extraction success are separate concerns.
- Always preserve debug artifacts when extraction fails:
  - final URL
  - page title
  - HTML snapshot
  - screenshot when useful
- Prefer fallback selectors over one brittle selector.
- Keep parsing logic isolated and testable.

### Anti-brittleness
- Prefer signals like aria-labels and semantic text over deeply nested class selectors where possible.
- Add fallbacks before widening concurrency.
- Validate crawler changes with one watch and low concurrency before scaling up.

## Batch Processing Rules

- Batch execution must be safe to run under cron.
- A single watch failure should be recorded and isolated.
- Batch summary must clearly show:
  - total watches
  - success count
  - failed count
  - warning count
  - duration
- Update `last_checked_at` only for successful watches.
- Do not mark a run successful before persistence and alert evaluation are complete.

### Concurrency
- Default to conservative concurrency.
- Increase concurrency only after confirming stability.
- Prefer correctness and survivability over raw speed.

## Monitoring and Alerting Rules

MVP alert rules:
- `BELOW_TARGET`
- `NEW_LOW_7D`

### Principles
- Alerting is monitoring logic, not scraping logic.
- Evaluate alerts only after normalized offers are persisted.
- Use the cheapest normalized offer for rule evaluation.
- Keep alert rule evaluation pure and testable.
- Persist alert events in `app.flight_alert_events`.

### Future improvements
- add cooldown and duplicate suppression
- add strong-drop detection
- add alert delivery channels
- add route-level adaptive frequency

## Testing Expectations

When modifying code, cover the highest-risk layer first.

### Always consider tests for
- parsing logic
- alert rule evaluation
- repository queries
- service orchestration that combines DB reads and writes

### Minimum checks before proposing a change
- `uv run pytest -v`
- `uv run ruff check src tests`
- `uv run mypy src`

If tests are not added, explain why the change is low-risk or how the existing tests still protect behavior.

## Debugging Workflow

When the worker fails:
1. reduce concurrency
2. run a single watch
3. inspect final URL
4. inspect debug HTML
5. inspect parsed offer output
6. verify DB writes
7. only then expand back to batch scale

When batch output looks wrong:
1. verify fetch run creation
2. verify normalized offers exist
3. verify monitoring query returns expected cheapest offer
4. verify alert rule inputs
5. verify alert event persistence

## Refactoring Preferences

Prefer these changes:
- extract helpers
- isolate repositories
- isolate evaluator logic
- reduce coupling between ingestion and app-specific logic

Avoid these changes unless clearly justified:
- large rewrites
- broad renames across many modules
- adding frameworks or libraries for minor problems
- premature abstractions

## Output Expectations for Claude

When asked to implement changes in this repository:
- think in small, reviewable steps
- state the next step before jumping ahead
- preserve architecture boundaries
- prefer production-minded but simple solutions
- assume the repository should evolve into a more robust DE pipeline over time

When suggesting code:
